A Priority Parallel Execution Framework in Java

发表于 2017-11-19

In our system, we have a service (FooService) that tells us the preference of a customer. This service depends on several other services’ output to aggregate the perference in the following way.

The initial version

First, we have a PerferenceService interface and every service (Service1-5 and FooService itself) implements this interface to return the customer perference in getPreference method.

In our FooService, it simply call Service1 - Service5 one by one. Once there is a service that returns a value, FooService stops calling the rest services and returns the value.

In this way, the DEPENDENCIES list is also a priority list. If Service1 can tell us the perference, we’ll make it the final result and will not make all other service calls.

However, after running the FooService for some time, we monitored that Service2 and Service4 has a longer latency than we expected. We’d like to call Servcie2 and Service4 in parallel to make our FooService faster. But the priority does not change. Once we get a result from Service2, we don’t want to waste our time waiting for service4 to return. But in case Service2 cannot tell us the customer’s perference, we still have Service3, Service4 and Service5.

If we use merge the 2 stpes like the following way, actually we are still calling each service one by one. (Make service calls in CompletableFuture.supplyAsync, wait for the service result in CompletableFuture::join, then move to the next service call)

However, it’s not a perfect world. We know this will not work because Service3 and Service5 are somehow traffic sensitive services. They cannot survive the increased traffic if we make a call to them for every customer. Plus, it does not make sense if we call Service5 everytime. We only need Service5 when Service1 - Service4 all failed to tell us the customers perference. It’s a waste to make such uncessary calls.

The final version

Before we move to the next solution, let’s summarize the requirement here:

  • Our FooService depends on Service1 - Service5 to get the customer perference
  • Service1 - Service5 has the priority that we only return the perference from ServiceN+1 when ServiceN cannot tell us the customer perference
  • There are some services call that take too long time that we want to call them in parallel (Service2 and Service4 in this case)
  • There are some services that are traffic sensitive that we should only call them when we cannot get the result from all higher priority services

CompletableFuture.supplyAsync can accept a second parameter which is an ExecutorService instance. In real world systems, we usually need to create a thread pool and pass it to supplyAsync. The default one usually is too samll for us.
